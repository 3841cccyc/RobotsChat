## Executive Summary

## 2. @提及机制的实现

### 2.1 @提及的识别与解析

@提及机制是群聊机器人的核心交互方式。实现要点:

```python
import re

# @提及的正则匹配模式
MENTION_PATTERN = r'@(\w+)'

def parse_mentions(message: str) -> list[str]:
    """解析消息中的@提及"""
    return re.findall(MENTION_PATTERN, message)

def contains_mention(message: str, bot_name: str) -> bool:
    """检查消息是否@了特定机器人"""
    mentions = parse_mentions(message)
    return bot_name in mentions or f"@{bot_name}" in message
```

### 2.2 @提及响应策略

| 策略 | 行为 | 实现难度 |
|------|------|----------|
| **必须回复** | 被@时强制响应 | Low |
| **优先回复** | 被@时提高响应优先级 | Medium |
| **智能判断** | 被@时结合上下文决定是否回复 | High |
| **忽略@** | 机器人可选择不回复@消息 | Low |

### 2.3 @提及上下文构建

被@时的上下文应包含:

1. **原始消息**: 包含@的完整消息
2. **发送者信息**: @机器人的用户身份
3. **对话历史**: 最近N轮对话(上下文窗口)
4. **意图识别**: 判断用户是想提问、命令还是闲聊

### 2.4 项目当前实现分析

当前项目通过`_get_bot_context`方法实现上下文构建,但**尚未实现@提及机制**。建议扩展:

```python
def _handle_mention(self, message: str, bot_name: str, all_messages: list) -> bool:
    """处理@提及,返回是否应该响应"""
    if not contains_mention(message, bot_name):
        return False  # 没有@该机器人

    # 可以添加:是否必须回复的配置
    return True  # 默认必须回复
```

## 3. 短消息连续输出设计模式

### 3.1 短消息拆分的原因

- **用户体验**: 长消息一次性展示阅读压力大
- **平台限制**: 部分平台有单条消息长度限制
- **交互节奏**: 短消息更符合自然对话节奏
- **减少刷屏**: 分散输出降低对其他用户的干扰

### 3.2 项目当前实现

项目已实现完整的短消息拆分机制:

#### 3.2.1 去重处理 (`_deduplicate_and_split`)

```python
def _deduplicate_and_split(self, text: str, previous_content: str = "") -> tuple:
    """
    去除重复内容并拆分:
    1. 去除与之前内容重复的部分
    2. 去除连续重复的短语(如 "你好呀！你好呀！")
    3. 去除句内重复
    4. 判断是否需要拆分(长度>80字符)
    """
```

#### 3.2.2 消息拆分 (`_split_into_short_messages`)

```python
def _split_into_short_messages(self, text: str, max_length: int = 80) -> list[str]:
    """
    将长文本拆分成多个短消息:
    - 断点标记: 。！？；，.!?;
    - 每30字符遇到断点则拆分
    - 超过80字符强制拆分(优先在空格处)
    """
```

### 3.4 推荐的短消息输出模式

#### 模式A: 即时拆分模式
```
输入: "这是一个很长的回复..."
输出:
  [消息1] "这是"
  [延迟0.5s]
  [消息2] "一个很长"
  [延迟0.5s]
  [消息3] "的回复..."
```

#### 模式B: 缓冲拆分模式
```
1. 收集完整响应
2. 后处理去重+拆分
3. 分批发送拆分后的消息
4. 每条消息之间添加延迟
```

#### 模式C: 智能拆分模式
```
1. 实时分析输出内容
2. 在自然断点(句号、段落)处拆分
3. 立即发送已拆分的片段
4. 剩余内容继续生成
```

## 5. MVP 推荐

### 必须实现 (Table Stakes)

1. **顺序发言模式** - 已实现
2. **流式响应输出** - 已实现
3. **消息去重拆分** - 已实现

### 建议扩展 (Differentiators)

1. **@提及机制** - 高价值,建议优先实现
2. **相关激活策略** - 已实现(策略5)
3. **关键词触发** - 可丰富交互形式
4. **打字动画** - 提升用户体验

### 不建议实现 (Anti-Features)

| 功能 | 原因 |
|------|------|
| 无限长度消息 | 违反短消息设计原则 |
| 即时无延迟输出 | 缺乏对话自然感 |
| 所有消息必回 | 应支持选择性回复 |

## 6. 总结

项目已具备群聊机器人的核心功能,包括流式输出、消息拆分去重、自动多轮聊天等。在当前基础上,建议优先实现@提及机制,这将是群聊机器人最重要的交互入口。短消息连续输出已实现完整,但可考虑增加打字动画等视觉反馈。
